## NasDeck 介绍

告别繁琐配置，NasDeck 让您轻松拥有专属的家庭智能娱乐中心！ 作为一站式自动化编排工具，它提供海量模板，只需一键即可唤醒家庭娱乐中心，让一切尽可能在你的掌握之中。

## 导航设计

整体导航分为 5 个一级模块，每个模块下包含子页面或子功能：

| 一级导航 | 路由 | 说明 |
|---------|------|------|
| 服务概览 | `/dashboard` | 仪表盘首页，全局状态一览 |
| 服务编排 | `/services` | NAS 服务卡片管理与编排 |
| 自动化管理 | `/automation/*` | 按媒体类型划分的自动化配置 |
| Docker | `/docker/*` | 容器编排/管理、镜像、主机 |
| 系统设置 | `/settings` | 用户、通知渠道、插件管理 |

**实现要点**：

- 侧边栏导航组件（`Sidebar`）需根据路由高亮当前模块，支持折叠/展开子菜单
- 自动化和 Docker 模块有二级导航，使用嵌套路由 + `<Outlet />` 渲染
- 移动端适配：侧边栏收起为抽屉式，顶部显示汉堡菜单

#### 服务概览

仪表盘首页，提供 NAS 服务的全局状态总览和快捷操作入口。

**功能描述**：
- **状态统计卡片**：显示已接入服务数、活跃容器数、今日更新数、异常告警数
- **服务健康面板**：展示各 NAS 服务（Jellyfin/Komga/MoviePilot 等）的连接状态、最近同步时间
- **最近更新流**：按时间线展示最近的订阅更新（来自 UpdateLog）
- **快捷操作**：一键检查更新、快速添加订阅、跳转到常用服务

**实现要点**：
- 后端新增 `GET /api/dashboard/summary` 聚合接口，一次返回所有统计数据，避免前端并发多个请求
- 状态统计通过 SQL 聚合查询（COUNT 已启用服务、RUNNING 容器、今日 UpdateLog、异常状态）
- 服务健康检查复用插件的 `test_connection()` 方法，结果可缓存 5 分钟
- 前端使用 React Query 的 `refetchInterval` 实现数据自动刷新（60 秒间隔）
- 更新流使用已有的 `UpdateLog` 模型，按 `created_at DESC` 分页取最近 20 条

#### 服务编排

以卡片形式管理所有已接入的 NAS 服务，支持添加新服务和编排服务依赖关系。

**功能描述**：
- **服务卡片列表**：每个已配置的插件实例（PluginInstance）显示为一张卡片，展示服务名称、类型图标、连接状态、关联订阅数
- **添加服务**：弹窗中选择服务类型（Jellyfin/Komga/MoviePilot 等），填写连接配置（URL/API Key 等），测试连接后保存
- **服务详情**：点击卡片进入详情页，展示该服务下的数据源、订阅列表、更新记录
- **服务编排**：支持设置服务启动顺序和依赖关系（如 MoviePilot 依赖 Jellyfin），编排信息保存在配置中

**实现要点**：
- 复用现有 `PluginInstance` 模型和 `/api/plugins/` 端点，前端以卡片视图重新呈现
- 新增 `GET /api/plugins/instances/{id}/sources` 和 `/api/plugins/instances/{id}/items` 端点，暴露插件的 `get_sources()` 和 `get_items()` 方法，供服务详情页调用
- 新增 `POST /api/plugins/instances/{id}/test-connection` 端点，前端在添加服务时先测试连接
- 服务编排的启动顺序存储为 JSON 配置（新增 `ServiceOrchestration` 模型或复用 `PluginInstance` 的 config 字段）
- 前端卡片组件使用 `dnd-kit` 实现拖拽排序，编排顺序持久化到后端

### 自动化管理

自动化需要单独为以下几个提供自动化配置，需要做到用户在服务编排选择配置后，自动启动，自动完成配置，也支持服务接管，为原来部署的服务提供配置接管。

#### 自动化流程（统一）

所有媒体类型遵循相同的自动化流程：

```
服务发现 → 配置模板 → 自动部署 → 状态监控 → 服务接管
```

1. **服务发现**：扫描 Docker 环境中已运行的媒体服务容器，识别服务类型（基于镜像名或容器标签）
2. **配置模板**：每种媒体类型提供预设配置模板（端口映射、存储路径、环境变量），用户可微调后一键应用
3. **自动部署**：根据配置模板自动拉取镜像、创建容器、启动服务、初始化插件实例
4. **状态监控**：部署完成后自动进入监控状态，检测服务是否正常启动，失败时自动重试（最多 3 次）
5. **服务接管**：对已手动部署的服务，通过 Docker 标签或端口扫描发现后，导入配置到平台管理

#### 各类型详情

1. **影视**

   - 推荐服务：Jellyfin / Emby / Plex
   - 自动化内容：媒体库扫描路径配置、字幕插件安装、硬件转码检测与配置
   - 实现要点：通过 Jellyfin API `/Library/VirtualFolders` 自动创建媒体库；检测 `/dev/dri` 判断硬件加速可用性

2. **漫画**

   - 推荐服务：Komga / Kavita
   - 自动化内容：漫画库目录挂载、OPDSFeed 配置、元数据刮削
   - 实现要点：Komga 通过 Basic Auth 自动完成初始用户创建和库路径绑定

3. **书籍**

   - 推荐服务：Calibre-Web / Kavita
   - 自动化内容：书籍库目录初始化、格式转换服务配置、阅读进度同步
   - 实现要点：Calibre-Web 需要预创建 metadata.db；支持自动导入 epub/mobi/pdf

4. **音乐**

   - 推荐服务：Navidrome / MusicBrainz
   - 自动化内容：音乐库扫描、封面自动下载、播放列表同步
   - 实现要点：Navidrome 首次启动自动扫描；需配置 LASTFM_API_KEY 实现歌词和推荐

5. **游戏**

   - 推荐服务：RomM / Steam ROM Manager
   - 自动化内容：ROM 目录结构初始化、元数据刮削、模拟器配置
   - 实现要点：RomM 支持自动识别平台（NES/SNES/GBA 等）并匹配封面；需配置 IGDB API 凭据

6. **图库**

   - 推荐服务：Immich / PhotoPrism
   - 自动化内容：照片备份目录配置、AI 人脸识别启用、相册自动分类
   - 实现要点：Immich 的 machine-learning 容器需要独立部署；PhotoPrism 需配置 TensorFlow 加速

7. **资讯**

   - 推荐服务：FreshRSS / Miniflux / TT-RSS
   - 自动化内容：RSS 源订阅导入、定时抓取频率配置、全文内容提取
   - 实现要点：支持 OPML 文件一键导入导出；通过 Fever API 兼容移动端阅读器

#### 实现要点（通用）

- 每种媒体类型对应一个「自动化配置器」类（`BaseAutomationConfigurator`），统一接口：`discover()` / `deploy(config)` / `import_existing(config)` / `health_check()`
- 配置模板使用 JSON Schema 定义（与插件系统的 `config_schema` 复用同一机制），前端动态渲染表单
- 自动部署产生的容器自动打上 `managed-by=nasdeck` 标签，用于区分用户手动创建和平台管理的容器
- 服务接管时需处理端口冲突检测和存储卷映射冲突
- 所有自动化操作记录审计日志（`AutomationLog` 模型），支持回滚

### Docker

docker下面需要分为以下几个导航，提供全面的docker服务管理配置功能。

#### 1. 容器编排

管理和编排容器组的生命周期，类似轻量级 docker-compose。

**功能描述**：
- **编排列表**：显示所有编排组（Stack），每个 Stack 包含多个关联容器
- **创建编排**：通过 YAML 编辑器或可视化表单定义编排，指定服务依赖和启动顺序
- **生命周期管理**：一键启动/停止/重启整个 Stack，按依赖顺序执行
- **模板市场**：预置常用 NAS 服务的编排模板（如 Jellyfin + Jellyfin-ffmpeg + 配置管理器）

**实现要点**：
- 后端新增 `Stack` 和 `StackService` 模型，存储编排定义（YAML/JSON）
- 编排引擎解析服务依赖图（有向无环图），生成启动/停止序列
- 使用 Docker SDK 的 `containers.create()` + `containers.start()` 替代 docker-compose 调用
- YAML 编辑器前端使用 CodeMirror 6，支持语法高亮和校验
- 模板市场初始版本内置模板，后续可扩展为从 GitHub 拉取

#### 2. 容器管理

管理单个 Docker 容器的生命周期和资源。

**功能描述**：
- **容器列表**：展示所有容器（运行中/已停止），显示名称、镜像、状态、CPU/内存占用、运行时长
- **容器操作**：启动、停止、重启、删除、查看日志
- **资源监控**：实时显示容器的 CPU、内存、网络 I/O、磁盘 I/O（基于 Docker Stats API）
- **终端接入**：通过 Web Terminal 进入容器内部（基于 xterm.js + WebSocket）

**实现要点**：
- 复用现有 `DockerManager` 和 `/api/docker/containers/` 端点
- 资源监控通过 `container.stats(stream=False)` 获取，前端轮询 5 秒间隔
- Web Terminal 需后端新增 WebSocket 端点 `/ws/container/{id}/terminal`，使用 `docker exec -it` 创建交互式会话
- 容器列表支持按状态筛选（运行中/已停止/全部）和搜索
- 分页加载，单页 20 条，避免容器数量多时前端卡顿

#### 3. 镜像

管理 Docker 镜像的拉取、查看和清理。

**功能描述**：
- **镜像列表**：展示所有镜像，显示仓库名、标签、大小、创建时间、关联容器数
- **拉取镜像**：输入镜像名和标签，显示拉取进度（层下载进度）
- **镜像详情**：查看镜像层（Layer）信息、暴露端口、环境变量、挂载点
- **清理镜像**：一键清理悬空镜像（dangling）、按时间范围清理旧镜像

**实现要点**：
- 后端新增镜像管理端点：`GET /api/docker/images`、`POST /api/docker/images/pull`、`DELETE /api/docker/images/{id}`
- 拉取进度通过 `docker.api.image.pull()` 的流式响应解析，前端用 SSE（Server-Sent Events）实时展示
- 清理使用 `docker.images.prune()` API，支持 `dangling_only` 和 `until` 参数
- 镜像大小格式化为人类可读单位（MB/GB）

#### 4. 主机

管理 Docker 主机本身的资源和配置。

**功能描述**：

- **系统信息**：显示 Docker 版本、操作系统、内核版本、运行时间
- **资源总览**：CPU 核心数/使用率、总内存/已用内存、磁盘总容量/已用容量
- **Docker 配置**：存储驱动、日志驱动、镜像加速器配置、数据根目录
- **事件流**：实时展示 Docker 引擎事件（容器创建/销毁、镜像拉取等）

**实现要点**：

- 后端新增 `GET /api/docker/host/info` 和 `GET /api/docker/host/stats` 端点
- 系统信息通过 `docker.info()` 和 `docker.version()` 获取
- 资源使用率通过 `psutil` 库获取宿主机指标（Docker SDK 不提供宿主机资源信息）
- Docker 事件使用 `docker.events()` 流式监听，通过 WebSocket 推送到前端
- 前端使用仪表盘式图表展示资源占用（环形图 + 进度条）
